## Recommendations

1. **`daemon_api_keys(account_id)` index** — 114k seq scans at 0.1% idx usage. The `account_id TEXT` column is missing an index. The #891 erasure cascade Step 4a runs `DELETE FROM daemon_api_keys WHERE account_id = ANY($client_ids)` — this will be a full table scan without the index. File a P1 index ticket; address in the #891 migration as a same-PR fix.

2. **`accounts` index audit** — 22k seq scans, 0% idx usage. Likely missing index on `client_id TEXT` or `user_id`. #891 Step 0 reads `SELECT client_id FROM accounts WHERE id=$account_id` (PK lookup, fine) but any WHERE-by-client_id path will scan. File P2 index audit ticket.

3. **`users` index audit** — 60k seq scans, 4.9% idx usage. Likely missing index on `clerk_id` or `email` for auth lookups. File P2 ticket.

4. **`set_cards` DISTINCT ON tuning** — 367ms mean over 5 recurring calls. A composite index supporting the DISTINCT ON order could reduce latency significantly. File P3 tuning ticket.

**Tickets to route through Pam:**
- `daemon_api_keys(account_id)` index — P1 (directly affects #891 cascade + daemon auth path)
- `accounts` index audit — P2
- `users` index audit (clerk_id/email) — P2
- `set_cards` DISTINCT ON query tuning — P3
